# django-apcore v0.3.0 Redesign — Extension-First Architecture

> **Date**: 2026-02-23
> **Status**: Approved
> **Scope**: Full redesign of django-apcore to leverage apcore v0.6.0 + apcore-mcp v0.4.0

---

## 1. Context

### 1.1 Why Redesign

The upstream libraries have had major releases:

- **apcore**: v0.5.0 → v0.6.0 — Extension System, AsyncTaskManager, CancelToken, W3C TraceContext, enhanced Registry (discoverer/validator hooks, hot-reload, describe), streaming annotations, new error types
- **apcore-mcp**: v0.3.0 → v0.4.0 — requires apcore>=0.6.0, Prometheus metrics endpoint, trace_id passback, input validation, schema integration

django-apcore v0.2.0 was designed around apcore v0.5.0's simpler API surface. The v0.6.0 Extension System fundamentally changes how components should be composed — from manual singleton wiring to declarative extension registration and auto-assembly.

### 1.2 Design Principles

1. **Extension-First**: Use apcore's `ExtensionManager` as the primary composition mechanism
2. **Django-Native**: Express all configuration through Django settings, all operations through management commands
3. **Thin Wrapper**: Delegate protocol logic to apcore, transport to apcore-mcp, schema generation to django-ninja/drf-spectacular
4. **Full Feature Parity**: Expose all apcore v0.6.0 + apcore-mcp v0.4.0 capabilities through Django-idiomatic APIs

## 2. Architecture

### 2.1 Core Idea

django-apcore v0.3.0 positions itself as **the Django adapter layer for apcore's Extension System**:

- Implements `Discoverer` protocol → `DjangoDiscoverer`
- Implements `ModuleValidator` protocol → `DjangoModuleValidator`
- Implements `ContextFactory` protocol → `DjangoContextFactory` (with W3C TraceContext)
- Configures middleware, ACL, span exporters through Django settings → registers them as extensions
- Uses `ExtensionManager.apply(registry, executor)` for auto-assembly

### 2.2 Startup Flow

```
Django startup
  → ApcoreAppConfig.ready()
    → 1. Validate all APCORE_* settings
    → 2. setup_extensions(settings) → ExtensionManager
         ├── DjangoDiscoverer → ext_mgr.register("discoverer", ...)
         ├── DjangoModuleValidator → ext_mgr.register("module_validator", ...)
         ├── User-configured middlewares → ext_mgr.register("middleware", ...)
         ├── User-configured ACL → ext_mgr.register("acl", ...)
         └── User-configured span_exporter → ext_mgr.register("span_exporter", ...)
    → 3. Create Registry(config=...) and Executor(registry, ...)
    → 4. ext_mgr.apply(registry, executor) → auto-assembly
    → 5. registry.discover() → DjangoDiscoverer finds modules
         ├── Scans APCORE_MODULE_DIR for *.binding.yaml → BindingLoader
         └── Scans INSTALLED_APPS for apcore_modules → @module decorators
    → 6. (Optional) Setup hot-reload watching
    → 7. (Optional) Start embedded MCP server
    → 8. (Optional) Initialize AsyncTaskManager
```

## 3. Component Design

### 3.1 extensions.py — Extension Adapter Layer

**DjangoDiscoverer** (implements `Discoverer` protocol):
- Scans `APCORE_MODULE_DIR` for `*.binding.yaml` (configurable via `APCORE_BINDING_PATTERN`) → loads via `BindingLoader`
- Scans `INSTALLED_APPS` for `apcore_modules` submodules → discovers `@module` decorated functions
- Returns `list[dict[str, Any]]` per the Discoverer protocol

**DjangoModuleValidator** (implements `ModuleValidator` protocol):
- Validates module_id format (not in RESERVED_WORDS, within MAX_MODULE_ID_LENGTH)
- Validates input/output schemas are valid JSON Schema
- Returns `list[str]` of error messages (empty = valid)

**setup_extensions(settings: ApcoreSettings) → ExtensionManager**:
- Creates ExtensionManager
- Registers DjangoDiscoverer as "discoverer"
- Registers DjangoModuleValidator as "module_validator"
- Registers user-configured middlewares from APCORE_MIDDLEWARES
- Registers ACL from APCORE_ACL_PATH if configured
- Registers span exporter from APCORE_TRACING if configured
- Returns configured ExtensionManager ready for apply()

### 3.2 context.py — Enhanced DjangoContextFactory

```python
class DjangoContextFactory:
    def create_context(self, request: Any) -> Context:
        # Extract W3C traceparent from request headers
        trace_parent = TraceContext.extract({
            "traceparent": request.META.get("HTTP_TRACEPARENT", "")
        })

        # Build Identity from Django user
        identity = self._build_identity(request.user)

        # Create Context with trace_parent
        return Context.create(
            identity=identity,
            trace_parent=trace_parent,
        )

    def _build_identity(self, user) -> Identity:
        if hasattr(user, "is_authenticated") and user.is_authenticated:
            return Identity(
                id=str(user.pk),
                type="user",
                roles=tuple(user.groups.values_list("name", flat=True)),
                attrs={"username": user.username},
            )
        return Identity(id="anonymous", type="anonymous", roles=())
```

Key change: `Identity.roles` is now `tuple[str, ...]` (frozen) per apcore v0.6.0.

### 3.3 registry.py — Simplified Singleton Management

The singletons are preserved but simplified. ExtensionManager handles wiring:

```python
_registry: Registry | None = None
_executor: Executor | None = None
_ext_manager: ExtensionManager | None = None
_lock = Lock()

def get_extension_manager() -> ExtensionManager:
    """Get or create the ExtensionManager singleton."""
    ...

def get_registry() -> Registry:
    """Get or create the Registry singleton, configured via ExtensionManager."""
    ...

def get_executor() -> Executor:
    """Get or create the Executor singleton, configured via ExtensionManager."""
    ...
```

The key simplification: instead of manually wiring discoverer, validator, ACL, middleware, and span exporter, we call `ext_mgr.apply(registry, executor)` once.

### 3.5 shortcuts.py — Expanded Convenience API

Existing (preserved):
- `executor_call(module_id, inputs, context=None)` — sync execution
- `executor_call_async(module_id, inputs, context=None)` — async execution
- `executor_stream(module_id, inputs, context=None)` — streaming execution
- `report_progress(context, progress, total=None, message=None)` — progress reporting
- `elicit(context, message, schema=None)` — user input elicitation

New additions:
- `cancellable_call(module_id, inputs, context=None, timeout=None)` — sync execution with CancelToken, auto-cancel on timeout
- `cancellable_call_async(module_id, inputs, context=None, timeout=None)` — async version
- `submit_task(module_id, inputs, context=None)` → `str` (task_id) — submit to AsyncTaskManager
- `get_task_status(task_id)` → `TaskInfo` — query task status
- `cancel_task(task_id)` → `bool` — cancel a running task

### 3.7 apps.py — Extension-First AppConfig

The `ready()` method is rewritten to use the Extension-First startup flow:

1. Validate all settings
2. Build ExtensionManager via `setup_extensions()`
3. Create and configure Registry + Executor via `ext_mgr.apply()`
4. Run `registry.discover()`
5. Optionally set up hot-reload watching
6. Optionally start embedded MCP server
7. Optionally initialize AsyncTaskManager

## 5. Breaking Changes from v0.2.0

1. **Minimum apcore version**: 0.5.0 → 0.6.0
2. **Minimum apcore-mcp version**: 0.3.0 → 0.4.0
3. **Identity.roles**: list[str] → tuple[str, ...] (follows upstream change)
4. **Internal startup**: Manual singleton wiring → ExtensionManager.apply()
5. **New settings**: 13 new APCORE_* settings added

Migration path: Users only need to update dependency versions. No public API changes are breaking. The Identity.roles change is transparent since DjangoContextFactory already constructs Identity internally.

## 7. Non-Goals (Unchanged)

- Django Auth/Permission ↔ apcore ACL mapping
- Django Middleware ↔ apcore Middleware bridging
- Django Admin integration
- Pure Django views scanner
- Database models or migrations
- Custom MCP transport implementation
